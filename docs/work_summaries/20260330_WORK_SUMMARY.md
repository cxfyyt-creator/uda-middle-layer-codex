## 2. CMG GEO 案例的重新转换与运行问题分析

今天较大一块工作都围绕 `inputs/cmg/geo/` 这组案例展开。先后做了以下事情：

1. 重新批量转换了 GEO 目录下全部 `mxgeo*.dat`；
2. 排查了 `mxgeo004`、`mxgeo006` 之前被校验挡住的原因；
3. 排查了 `mxgeo008`、`mxgeo010`、`mxgeo012` 这类 `FLXB-IN` 链式案例为什么仍然不能直接运行。

### 2.1 `mxgeo004` 和 `mxgeo006` 之前为什么不能跑

这两个文件的问题不是原始 CMG deck 天生不能跑，而是它们在走“标准化校验链路”时被中间层挡住了。具体来说，标准模型中的 `reservoir.porosity` 校验当前要求孔隙度满足 `(0, 0.60]`，而 `mxgeo004.dat` 和 `mxgeo006.dat` 都包含大量 `0.0` 孔隙度格块。结合原始 deck 注释可知，这两组案例本来就属于带有 `zero value porosity grids` / `null blocks` 的示例，因此这属于“标准化校验规则不兼容此类案例”，而不是 CMG 原始 deck 一定错误。

### 2.2 `mxgeo008` / `mxgeo012` 为什么仍然不能直接跑

今天也专门分析了 `FLXB-IN` 依赖问题。结论是：当前真正的问题不是 `.dat` 没有生成，而是 **下游案例引用的 flux-boundary 文件名，与上游实际跑出来的文件名不一致**。例如：

- `mxgeo008_converted.dat` 里仍然写的是 `*FLXB-IN 'mxgeo007.flxb'`；
- `mxgeo012_converted.dat` 里仍然写的是 `*FLXB-IN 'mxgeo011.flxb'`；
- 但目录里实际由上游案例运行产生的是 `mxgeo007_converted.flxb`、`mxgeo011_converted.flxb`。

所以本质上是“依赖链仍未对齐”。也就是说：

- 上游 case 已经跑出了 `.flxb`；
- 下游 case 仍然按原始名字查找；
- 名字不一致导致运行时找不到依赖。

今天针对这个问题只做了**分析**，没有继续沿着这条线展开长期方案，因为用户随后要求先不考虑 8/12 这部分问题。

### 2.3 相关临时代码调整与回退

在分析 `FLXB-IN` 问题时，曾短暂尝试过一种更宽松的 source-faithful 策略：即缺失运行时依赖时也先允许把 `.dat` 生成出来，避免在生成阶段直接拦住。但在进一步讨论后，用户明确要求先把这部分改动恢复，因此今天已将该放宽逻辑回退，恢复为：**source-faithful 生成阶段若缺少运行时依赖，仍由 preflight 阻断**。对应测试也已恢复并重新通过。

## 4. 关于真实工程文件的讨论：从案例视角转向工程视角

今天后半段最有价值的部分，是围绕“真实油藏模拟工程不是单文件，而是工程包”的讨论。用户指出：真实项目中，不会像示例案例那样把所有内容都放在一个小的 deck 里，实际大模型往往会把地质模型和大规模属性数据放在外部文件中。用户随后又进一步补充，以 CMG 为例，真实工程最典型的核心组合往往是：

- 一个主控 `dat`；
- 一个外部 `sip` 数据文件。

在这个前提下，今天形成了一个比较清晰的共识：

1. 你当前这套“中间层”本质上已经是 IR（统一中间表示），这一点不是未来才要建立，而是已经存在；
2. 真正要做的，不是“再设计一个新的 IR”，而是让现有中间层从 **案例型 IR** 升级成 **工程型 IR**；
3. 所谓工程型，不是推翻现有 `grid / fluid / wells / schedule` 这些语义块，而是要再补上一层“文件关系表达能力”。

换句话说，今天对未来规划的核心理解被压缩成了一句更简单的话：

> 现在的中间层更擅长描述“模型内容”；下一步要让它既能描述“模型内容”，也能描述“这些内容分别存在哪些工程文件里”。

## 5. 对未来规划的阶段性认识

基于今天的讨论，对后续路线形成了更明确的分层理解。

### 5.1 当前阶段的准确定位

你现在做的，不是“有没有 IR”的问题，而是“现有 IR 还停留在哪个层级”的问题。更准确地说，当前中间层已经能够支持：

- 解析；
- 转换；
- 校验；
- 生成；
- 基于示例 case 的往返验证。

但它目前仍更偏向“单文件 / 单 case / 字段驱动”的表达方式，还不足以很好地承载真实工程里的：

- 主文件与外部 payload 文件关系；
- 路径与依赖图；
- 运行时输入与运行时产物；
- 大规模网格与属性引用。

### 5.2 未来最合理的升级方向

今天最终比较认同的升级方向是：

- 不重做中间层；
- 不急着同时扩很多模拟器；
- 先把现有中间层升级成 **工程型 IR 1.0**。

这里的“工程型 IR 1.0”最关键的不是再补若干 keyword，而是补两种能力：

1. **FileRef / DataRef 能力**：让 IR 能表达“这个数据不是内联数组，而是来自某个外部文件”；
2. **Case Manifest 能力**：让 IR 能表达“一个工程由哪些文件组成、谁依赖谁、哪些是静态输入、哪些是运行时依赖”。

## 6. 今天形成的关键共识

今天最重要的几条共识如下：

1. 当前中间层已经可以被视为 IR，不必反复从零定义；
2. 真正的下一步不是“建立 IR”，而是“把现有 IR 从案例型升级成工程型”；
3. 对 CMG 而言，`dat + sip` 是非常典型、必须围绕其设计工程表达能力的真实场景；
4. GEO 中 `FLXB-IN` 问题说明，未来工程型表达里必须明确区分：
   - 静态输入依赖；
   - 运行时输入依赖；
   - 运行后生成产物；
5. 当前继续补案例仍然有价值，但它更应该逐步转向“回归测试集”的角色，而不该继续独占架构主线。
